🚫 BLOCKING Browser disconnect leaves _ssh_task orphaned with no cleanup path

At stake: If the user closes or refreshes the browser tab without sending vox_deactivate, _ssh_task continues running — SSH connection open, Whisper and TTS models loaded, audio threads alive. The plan's handle_ws cleanup only removes the connection from _conn_queues (no more output routing), but the task itself is never cancelled. Refreshing the browser would let the user send vox_activate again, starting a second _ssh_test task (the old one still holds the SSH connection). On a dev machine this silently accumulates zombie sessions.

Options:
  1. When handle_ws detects close/error and the connection count drops to zero, cancel _ssh_task — same code path as vox_deactivate. Tradeoff: if two browser tabs are open and one closes, the session dies. Acceptable given single-user design.
  2. Track a "last seen" timestamp per WS connection; have _activate_vox always cancel any existing task before creating a new one (already partially in the plan). Then leave orphaned tasks running until the next activation — at worst one zombie until the next vox_activate. Tradeoff: leaked resources until next user interaction.

⚠️ JUDGMENT CALL BG-thread error before ready.set() deadlocks the main thread forever

At stake: The new run_ui BG thread calls ready.set() only after site.start() succeeds. The current Tkinter code (line 264–265 of app.py) has a finally: if not ready.is_set(): ready.set() guard that unblocks the main thread even on error. The plan's pseudocode for the BG thread omits this guard — if site.start() raises (port conflict, missing dependency), the main thread waits on ready.wait(timeout=60.0) for a full 60 seconds before bailing out with a cryptic "failed to start" message.

Options:
  1. Preserve the finally: if not ready.is_set(): ready.set() guard from the current code. Add an error slot (e.g., a list the BG thread appends to) so the main thread can re-raise the exception. Tradeoff: a few extra lines; makes startup failures fast and debuggable.
  2. Keep ready.wait(timeout=60.0) as-is; rely on the 60s timeout to unblock. Tradeoff: a 60-second frozen window with no explanation if the port is taken.

⚠️ JUDGMENT CALL Synchronous thread joins inside _ssh_test's finally block block the event loop during cancellation

At stake: When _activate_vox cancels and awaits the previous _ssh_task, Python raises CancelledError into _ssh_test's finally block. That block calls trigger.close() (AppCommandTrigger), which does a synchronous self._thread.join(timeout=2.0) — a blocking call on the event loop thread. This is the same in the current Tkinter code, but cancellation via _activate_vox makes it a routine operation rather than a one-time shutdown. Each vox_deactivate → vox_activate cycle can stall the loop for up to 2 seconds, during which WebSocket messages queue up and the browser appears frozen.

Options:
  1. Wrap the blocking join in loop.run_in_executor(None, trigger.close) before the task is cancelled, or add a non-blocking close path to AppCommandTrigger. Tradeoff: more code; cleanly solves the hot path.
  2. Accept the 2-second stall — in practice vox_deactivate → vox_activate is a rare user action, and the UI shows "responding" state during the gap anyway. Tradeoff: occasional visible freeze; stall is bounded at 2s.

⚠️ JUDGMENT CALL vox --ui no longer auto-activates voice on launch — silent behavioral break

At stake: Currently, vox --ui starts the SSH connection and loads models immediately (the BG thread calls _ssh_test before ready.set()). After the rewrite, the session only activates when the browser sends vox_activate. A user who runs vox --ui and presses Beats before the browser finishes loading gets no recording — the hotkey event has no listener. This is a deliberate architectural change (lazy model load is the whole point), but the plan doesn't document it or update any help text. The verification checklist could pass without ever testing this cold-start path.

Options:
  1. Add a note in verification and in the panel's UI that the Beats button is only active after the browser sends vox_activate (e.g., a visible "Click ▶ to activate voice" prompt on load). Tradeoff: minor UI polish; prevents user confusion.
  2. Have run_ui optionally send a synthetic vox_activate on first WebSocket connection, mirroring the current eager-load behavior. Tradeoff: re-introduces eager model load; defeats lazy-load RAM goal.

ℹ️ INFORMATIONAL wt_settings_path default raises KeyError at import if LOCALAPPDATA is unset

At stake: The plan adds wt_settings_path: Path = field(default_factory=lambda: Path(os.environ["LOCALAPPDATA"]) / ...) to Config. os.environ["LOCALAPPDATA"] raises KeyError if unset (e.g., running as a service, in a minimal CI environment, or WSL). The existing fields use Path.home() which is always available. os.environ.get("LOCALAPPDATA", "") would produce a clearly wrong path that profiles.py can check rather than a hard crash at import time.

Options:
  1. Use os.environ.get("LOCALAPPDATA", "") — profiles.py already has a fallback for "file not found". Tradeoff: trivial change; keeps import side-effect-free.
  2. Leave as os.environ["LOCALAPPDATA"] — this is a Windows-only project and LOCALAPPDATA is always set in a real Windows desktop session. Tradeoff: fragile in non-interactive contexts.

What the plan handled well

  • Thread-safety constraints are explicit and correct. The "Critical constraints" section lists the exact invariants (selector loop, call_soon_threadsafe, broadcast-on-loop-thread only) and they match the existing codebase patterns precisely.
  • Per-connection async queue for WS sends is the right design. Avoiding await ws.send_str() in the TextDelta hot loop, with a maxsize queue and per-type overflow policy, correctly prevents backpressure from a slow browser tab stalling the audio pipeline.
  • Race-condition-free startup ordering is clearly specified. ready.set() after site.start() — pywebview can't open until the HTTP server is actually listening. This fixes a real race in many naïve implementations.
  • Port-conflict risk is pre-addressed. The plan explicitly calls out OSError on 7654 and mandates fail-fast with a clear message — no silent retry, no wrong-port pywebview.
  • 127.0.0.1 vs localhost CORS concern is caught proactively. Standardizing on 127.0.0.1 for both bind address and pywebview URL avoids a subtle WebView2 same-origin issue that would be hard to debug after the fact.
